## Angle (a) — Regression-eval tools: methodology + convergent guidance

**Headline: the industry converged on the same three rules — deterministic graders first, 20-50 golden cases from real failures, suite runs per config change at ~100% expected pass.**

- **Anthropic "Demystifying evals for AI agents"** (Tier B, refuter-confirmed verbatim, https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents): "deterministic graders where possible, LLM graders where necessary…, human graders judiciously"; "20-50 simple tasks drawn from real failures is a great start"; regression evals "should have a nearly 100% pass rate"; grade end-state not path; pass@k vs pass^k for consistency-critical behavior; read transcripts weekly; **eval-bug incident: Opus 4.5 scored 42% on CORE-Bench → 95% after fixing rigid grading** ("96.12" penalized vs expected "96.124991…") — suspect the eval before the config.
- **Anthropic API docs** (Tier B, https://platform.claude.com/docs/en/test-and-evaluate/develop-tests): grading order code > human ("avoid if possible") > LLM ("test to ensure reliability first"); "prioritize volume over quality"; judge model ≠ generation model; constrained output + reasoning-then-discard.
- **promptfoo** (Tier B, https://www.promptfoo.dev/docs/): 30+ deterministic asserts + 15+ model-graded; `llm-rubric` returns `{reason, score 0-1, pass}`; CI via GitHub Action + JUnit XML + pass-rate gate; no built-in semantic regression diff (parse results JSON). 22.1k stars.
- **Braintrust** (Tier B — **critic-corrected**: its golden-dataset encyclopedia page is QUALITATIVE, "start with a focused set of representative inputs"; the previously-quoted "20-50 examples" figure is NOT on that page — UNVERIFIED, dropped): cluster production failures before adding (un-clustered suites memorize — that claim stands, articles page). **LangSmith** (Tier B): start 10-20 high-quality examples; judges "require careful review of scores and prompt tuning". **OpenAI** (Tier B): meta-eval the grader against human ground truth before scaling. **DSPy** (Tier B/C): trainset 5-20 / devset ~50; assertions = retry-with-feedback. The 20-50 set-size convergence rests on **Anthropic (refuter-verified verbatim) + LangSmith 10-20** — it survives without Braintrust's number.

## Angle (b) — skill-creator plugin: exact mechanics (T1 — installed copy read on disk)

**Headline: a working, auth-free trigger-eval harness ships installed — but it tests a description PROXY and does not isolate the 20 real competing skills.**

- Two systems in `~/.claude/plugins/cache/claude-plugins-official/skill-creator/`:
  - **Capability evals**: `evals.json` (prompt + natural-language `expectations`) → session spawns paired subagents (with-skill vs baseline) → LLM grader agent → `python -m scripts.aggregate_benchmark` → mean±stddev (n−1) of pass_rate/time/tokens, default 3 runs/config; analyzer flags non-discriminating assertions and high-variance evals ("50% ± 40% — may be flaky"); HTML review viewer + optional blind A/B comparator (T1, SKILL.md + scripts).
  - **Trigger evals**: `[{query, should_trigger}]` (20 queries, near-miss negatives) → `scripts/run_eval.py` writes a **temporary COMMAND file** `.claude/commands/{skill}-skill-{uuid8}.md` with the description under test → `claude -p <query> --output-format stream-json`, 3 runs/query, threshold 0.5, 10 workers; `run_loop.py`: 60/40 stratified split, ≤5 iterations, `improve_description.py` via `claude -p` stdin (session auth — no API key, fix from PR #547 present locally), best-by-test-score, blinded history (T1, verbatim-grounded).
- **Confound (refuter-corrected)**: real installed skills compete with the temp eval command → false 0% trigger rates — **anthropics/skills issue #556** (https://github.com/anthropics/skills/issues/556); skill-hiding is only a PROPOSED fix there; **current `run_eval.py` contains no isolation logic** (refuter grep: nothing). In a 20-skill setup, absolute rates bias low; before/after deltas remain usable.
- Cost order: up to 20×3×5 = 300 headless `claude -p` calls per optimization run. Official CC docs recommend no eval harness; they cap skill descriptions at 1,536 chars and budget ~1% of context for skill listings (Tier B, https://code.claude.com/docs/en/skills).
- Own-docs limitation (T1, SKILL.md): "Subjective skills (writing style, design quality) are better evaluated qualitatively" — style/honesty behaviors are out of its assertion model. CLAUDE.md/output-style have **no native slot** in either system; the paired-run + grader + aggregate pattern is reusable off-label.

## Angle (c) — Published cases: what measurably works on agent behavioral config

- **Skill activation (the direct precedent)** — Scott Spence, sandboxed `claude -p` evals (Tier B, refuter-confirmed, https://scottspence.com/posts/measuring-claude-code-skill-activation-with-sandboxed-evals): baseline activation **50-55%** (12/22, 11/22 across runs); simple CLAUDE.md instruction ≈ no effect; **forced-eval UserPromptSubmit hook: 100% activation, 100% true negatives**; LLM-eval hook: 100% activation but **80% false positives** on edge cases (1/5 true negatives); **$5.59 / ~250 invocations**. Deterministic wiring beats LLM judgment — convergent with poneglyph's own 012 lesson and 017/US12 direction.
- **Instruction-file effectiveness — two Tier A studies, opposite axes**:
  - arXiv 2602.11988 ("Evaluating AGENTS.md", ETH Zurich, AGENTbench, 138 tasks, refuter-confirmed): LLM-generated context files **−3% success, +20-23% cost**; developer-written **+4%** — but "for all agents but Claude Code": **Claude Code gained ~0 even from developer-written files**. Agents follow the instructions; outcomes don't improve (largely redundant with repo docs).
  - arXiv 2601.20404 (124 PRs, refuter-confirmed): AGENTS.md present → **−28.64% median runtime, −16.58% output tokens**, comparable completion. Net: instruction files buy EFFICIENCY and specific-behavior compliance, not success rate.
- **Style/register testing**: IFEval pattern (arXiv 2311.07911) — code-verifiable instructions ("respond in Spanish", banned phrases, format position) without judges; PromptPex (arXiv 2503.05070, Tier B) auto-derives compliance tests from a prompt's stated rules. Aider publishes "percent using correct edit format" as a first-class prompt-compliance regression metric (Tier B, aider.chat/docs/leaderboards/).
- **Judge reliability for style criteria** (Tier A): MT-Bench GPT-4 judge >80% human agreement (arXiv 2306.05685) BUT position bias is real and judge/task-dependent — 150k+ instances, 15 judges; Position Consistency varies by judge and task (e.g., Claude-3.5-Sonnet 0.82 MTBench vs 0.76 DevBench; general+software-dev benchmarks, not code-specific) (**critic-corrected** wording — the earlier ">10% swing" headline number was not locatable in the paper; aclanthology 2025.ijcnlp-long.18). Verbosity/style bias 0.76-0.92, expert-domain agreement only 60-68%, and **intra-judge run-to-run reliability is low** (Rating Roulette, arXiv 2510.27106, EMNLP 2025, refuter-confirmed).

## Counter-evidence

1. **Generic "improvements" hurt** (Tier A, arXiv 2601.22025, refuter-confirmed): swapping task-specific prompts for generically "better" templates dropped extraction 100%→90% and RAG compliance 93.3%→80% while instruction-following scores ROSE — per-task golden prompts over generic style metrics.
2. **Context files can be net-negative** (Tier A, 2602.11988): −3% success +20% cost for LLM-generated ones; Claude Code gains nothing even from good ones (success axis).
3. **The famous "compliance decay curve" (95%→20-60% by message 6-10) is unmeasured folklore** (Tier D — verified by direct fetch of the source post: illustrative, no methodology).
4. **Goodhart/overfitting**: un-clustered failure-derived suites memorize history while near-duplicates pass (Tier B, Braintrust); production evals exist because benchmark gains "fail to translate" (OpenAI alignment).
5. **Judges are noisy instruments** (Tier A ×3: position bias, style bias, run-to-run inconsistency) — and the only in-domain measurement (Spence) shows 80% FP for an LLM-judge hook vs 0% for deterministic.
6. **Maintenance burden**: suite = "living artifact… clear ownership" (Anthropic); no published post-mortem of an abandoned prompt-eval suite found (absence, UNVERIFIED gap).

## Verdict

1. **Converged recipe** (B-tier unanimous + A-tier judge data): deterministic/code-graded checks first; single-call constrained-label rubric ONLY for irreducibly subjective criteria, judge ≠ generator, never pairwise; 20-50 golden prompts from real observed failures; run per config change; expect ≈100% pass; suspect the eval before the config.
2. **For poneglyph's exact pains, the measured precedents exist**: skill triggering (Spence harness + skill-creator's installed tooling, with the #556 isolation caveat), style/register compliance (IFEval-style code checks: Spanish, banned sycophancy phrases, BLUF position), CLAUDE.md changes (test efficiency tokens/runtime + named-behavior compliance — NOT overall success, which Tier A says won't move).
3. **Deterministic wiring beats LLM judgment in-domain**: 100%/0%FP (hook) vs 80%FP (LLM hook) — third independent confirmation of the 012/017 direction.
4. The installed skill-creator harness is usable TODAY for relative trigger deltas; absolute rates need skill isolation (#556).
5. Strongest guard rail: generic prompt "improvements" measurably degrade task performance while looking better on generic metrics — every eval must anchor to poneglyph's own named behaviors and real failures.
