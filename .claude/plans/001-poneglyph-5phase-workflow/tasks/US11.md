## User story

- **As a**: developer/Lead/skill en cualquier fase del workflow
- **I want**: una skill global invocable on-demand que aplique el patrón Socratic canónico sobre cualquier decisión, plan, output, o duda
- **So that**: no duplico el catálogo de preguntas en cada skill de fase y puedo invocar drillme en cualquier momento — incluso fuera de una fase formal (decisión ad-hoc, código mid-implementación, dilema de diseño)

## Acceptance criteria

- **AC1**: Given keywords `drill` / `drillme` / `socratic` / `5-whys` / `valida` / `cuestiona` / `challenge` / `antes de cerrar` en el prompt, when el Lead procesa, then `drillme` se auto-activa via `description`.
- **AC2**: Given el command `/drillme [contexto]`, when se invoca, then la skill produce ≥3 preguntas Socratic etiquetadas con su categoría canónica (`[location]`, `[approach]`, `[context]`, `[failure]`), ordenadas por relevancia al contexto.
- **AC3**: Given la skill activa **sin contexto explícito**, when detecta una fase activa (presencia de `spec.md`/`tasks/`/`tests.md`/etc. recientes en `.claude/plans/`), then carga las preguntas específicas de esa fase desde la US correspondiente (US2-US7) **además** del catálogo canónico.
- **AC4**: Given respuestas a las preguntas, when la skill las recibe, then aplica un check de calidad: respuestas vacías/evasivas → re-iterar; respuestas concretas → cerrar drillme y reportar.
- **AC5**: Given el principio "no siempre más es más", when el contexto es trivial o ya resuelto, then la skill produce 1-2 preguntas focalizadas en lugar de las 4 categorías completas, declarándolo: "Drillme reducido por contexto autoexplicativo".
- **AC6**: Given la skill operativa, when las skills US2-US7 llegan a su drillme block, then la línea final del block dice: "Apply drillme skill for full Socratic check — see `.claude/skills/drillme/SKILL.md`". NO duplican el catálogo canónico.
- **AC7**: Given fiabilidad skill→skill probabilística (research 2026-05-28), when las skills US2-US7 invocan drillme y la invocación NO se dispara, then el Lead detecta el gap en su verificación post-skill y la dispara manualmente. La skill drillme es **guidance**, no gate (Commandment IV NO aplica aquí).

## Catálogo canónico (literal, copiar al SKILL.md)

```markdown
## Socratic catalog — 4 canonical categories

Source: [Socratic Prompt Method, Jaseci Labs 2026](https://blogs.jaseci.org/blog/2026/03/10/socratic-prompt-method/) + [Towards AI: The Socratic Prompt 2025](https://towardsai.net/p/machine-learning/the-socratic-prompt-how-to-make-a-language-model-stop-guessing-and-start-thinking).

| Tag | Category | Purpose | Template question |
|---|---|---|---|
| `[location]` | Challenge location | Architecture awareness | "Is this the right place / file / layer for this?" |
| `[approach]` | Challenge approach | Pattern reasoning | "Why this approach over <alternative>? What did we reject?" |
| `[context]` | Introduce context | Dependency discovery | "How does this interact with <X>? What does it touch?" |
| `[failure]` | Probe failure modes | Edge case robustness | "What happens if <edge case>? Worst-case scenario?" |

## Complementary patterns

- **5-whys**: Repeatedly ask "why?" until reaching the root cause (max 5 iterations).
- **First principles**: Strip the problem to its fundamental atoms; rebuild from there.
- **Inversion**: Ask "what if the opposite were true?" — explore the counterfactual.
```

## Workflow detallado de la skill

1. **Parsear contexto**:
   - Si `$ARGUMENTS` contiene un brief explícito → tomar ese contexto literal.
   - Si no, escanear conversación reciente (últimos turns) + state de `.claude/plans/` para detectar fase activa.
   - Si fase detectada → cargar preguntas específicas de esa fase de la US correspondiente (US2.drillme para fase 1, US3.drillme para fase 2, etc.).

2. **Aplicar catálogo canónico**:
   - Generar al menos UNA pregunta por cada categoría aplicable (`[location]`, `[approach]`, `[context]`, `[failure]`).
   - Si alguna categoría no aplica al contexto → declararlo honestamente ("Skipping [location] — N/A in this context").
   - Cobertura mínima: 3/4 categorías cubiertas. Si <3 → smell — el contexto no es decidible aún.

3. **Aplicar técnicas complementarias** (según contexto):
   - Si síntomas vs causa raíz no claros → 5-whys (max 5 iteraciones).
   - Si problema novedoso sin patrón previo → first principles.
   - Si decisión cerrada entre 2 opciones → inversión.

4. **Output**:
   - Lista numerada de 3-7 preguntas, cada una con su tag `[categoría]` y la técnica subyacente si aplica.
   - Si modo `--silent`: solo preguntas, sin meta-explicación.
   - Si modo default: preguntas + breve "Por qué cada una".

5. **Iterar (opcional)**:
   - Si el usuario responde → leer respuestas → aplicar check de calidad:
     - Respuesta vacía / "no sé" sin razón → re-iterar esa pregunta con follow-up.
     - Respuesta evasiva ("depende", "más o menos") → pedir concreción.
     - Respuesta concreta → marcar pregunta como cerrada.
   - Continuar hasta que ≥80% preguntas cerradas con respuesta concreta.

6. **Reportar**:
   - Resumen 1-2 líneas: qué se exploró, qué quedó abierto.
   - Si hay preguntas sin respuesta concreta → listar como "Open drillme questions" para que el Lead/usuario decida si bloquean o no.

## SIEMPRE rules

- **Honestidad**: si una categoría no aplica → declararlo (no inventar pregunta artificial).
- **No softening**: las preguntas Socratic no se suavizan ("¿quizás podría haber un caso edge?"); van directas ("¿qué pasa si X?").
- **Calibración**: ajustar número de preguntas al peso de la decisión (decisión trivial = 1-2 preguntas; decisión arquitectónica = 5-7).

## Commandments cubiertos

| # | Cómo |
|---|---|
| I | Honest sobre lo que no sabe; preguntas directas sin softening |

## Adaptación intra-fase

| Señal | Adaptación |
|---|---|
| Contexto trivial / decidido / autoexplicativo | 1-2 preguntas focalizadas; declarar "Drillme reducido por contexto" |
| Decisión arquitectónica multi-criterio | 5-7 preguntas + técnicas complementarias (5-whys + inversión) |
| Mid-implementation, código abierto | Focalizar en `[failure]` + `[context]` (modos de fallo + interacciones); skip `[location]` (ya hay archivo) |
| Cierre de fase formal | Cargar preguntas específicas de la fase desde la US correspondiente + complementar con catálogo canónico |
| Usuario invoca explícitamente `/drillme` con brief | Tomar brief literal, aplicar 4 categorías sobre ese brief |

## Casos edge

- **Edge 1**: Skill se auto-activa pero no detecta contexto → preguntar al usuario "¿Sobre qué quieres drillme?".
- **Edge 2**: Usuario invoca `/drillme` mid-implementación pero no hay decisión que tomar → declarar "No detecto decisión abierta. Brief opcional?".
- **Edge 3**: Skill US2-US7 invoca drillme pero la invocación falla (probabilística) → el Lead detecta el gap en su verificación post-skill y dispara `/drillme` manualmente. Aceptable porque drillme es guidance, no gate.
- **Edge 4**: Usuario responde con "no sé" honesto a varias preguntas → declarar "Drillme inconcluso, necesitas más research/contexto antes de decidir".

## Smell signals

- ⚠️ Si las respuestas del usuario son todas "depende" → la decisión no está madura; pausar.
- ⚠️ Si drillme genera siempre las mismas 4 preguntas genéricas → falta calibración al contexto; revisar workflow paso 1.
- ⚠️ Si las skills US2-US7 nunca invocan drillme aunque sus drillme blocks lo indiquen → la fiabilidad skill→skill es el bottleneck; considerar promover invocación a comando explícito en el SKILL.md.
- ⚠️ Si drillme tiene >10 preguntas para una decisión → over-engineering; calibración mal hecha.

## Verificación post-implementación

- Smoke: `/drillme "estoy dudando entre A y B"` → produce 3-5 preguntas con tags `[categoría]`.
- Smoke: invocar drillme mid-conversación tras cuestión ambigua del usuario → auto-activa.
- Verificar que las skills US2-US7 referencian `drillme` en sus drillme blocks (no duplican catálogo).
- `bun test ./.claude/hooks/` sigue 81/81 (no toca hooks).

## Nota sobre fiabilidad skill→skill (research 2026-05-28)

Esta skill está diseñada para ser invocada por otras skills (US2-US7) pero la invocación skill→skill **NO está garantizada al 100%** según [docs Anthropic](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/overview) + [issue #59968](https://github.com/anthropics/claude-code/issues/59968).

**Por qué eso es aceptable aquí**: drillme es **guidance**, no gate (no Commandment IV). Si una skill no lo invoca por probabilismo, el output de fase NO se corrompe — solo pierde el check Socratic adicional. El Lead, al verificar el output de la skill, detecta el gap y dispara `/drillme` manualmente.

**Patrón canónico de invocación desde otra skill** (incluir en US2-US7 SKILL.md):

```markdown
## Drillme — Phase N

<preguntas específicas de fase>

> **For full Socratic check, invoke the drillme skill** (catálogo canónico + 4 categorías + técnicas complementarias). See `.claude/skills/drillme/SKILL.md`.
>
> If drillme auto-activation doesn't fire, the Lead must invoke `/drillme "<context of this phase>"` manually before closing.
```
