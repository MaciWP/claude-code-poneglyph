## User story

- **As a**: Lead orchestrator que acaba de construir el workflow 5-fases
- **I want**: ejecutar el workflow USANDO el workflow sobre una mini-tarea (dogfooding)
- **So that**: valido empíricamente que funciona end-to-end ANTES de declararlo estable

## Acceptance criteria

- **AC1**: Given workflow completo (US1-US9 cerrados), when se invoca `/flow --standard "<mini-task>"` (mini-task acordada con usuario, ej: "documentar el workflow en `.claude/plans/README.md`"), then se ejecuta el flujo completo sin intervención manual extraordinaria.
- **AC2**: Given el dogfooding completado, when se produce `retro.md` del mini-feature, then identifica ≥1 fricción real (cumple AC4 del spec) + ≥1 promoción candidata.
- **AC3**: Given el retro del dogfooding, when se compara con la experiencia del meta-refactor (US1-US9), then los hallazgos se consolidan en `.claude/plans/001-poneglyph-5phase-workflow/retro.md` (retro del propio meta-refactor).
- **AC4**: Given el retro consolidado, when termina, then frontmatter de `spec.md` y `tasks/index.md` del meta-refactor pasan a `status: closed`.
- **AC5**: Given posibles ajustes detectados durante dogfooding (smells en alguna skill, fricciones en gates, etc.), when son blockers → propone HUs follow-up en sesión siguiente; when son menores → documenta en retro pero no bloquea cierre.

## Workflow

1. **Acordar mini-task** con usuario (≤30 complejidad). Sugerencias:
   - "Documentar el workflow en `.claude/plans/README.md`" (escribir doc).
   - "Añadir una skill X simple para probar" (creación de skill simple).
   - "Refactor de un archivo concreto" (refactor acotado).
2. **Invocar `/flow --standard <mini-task>`**.
3. **Observar end-to-end** sin intervención (salvo hard gates obligatorios):
   - ¿Fase 1 produce spec.md útil?
   - ¿Fase 2 atomiza correctamente?
   - ¿Fase 2.5 produce tests útiles?
   - ¿Fase 3 ejecuta sin fricciones?
   - ¿Fase 4 detecta algo real?
   - ¿Fase 5 propone promociones que tengan sentido?
4. **Registrar fricciones**: notas durante el flujo de cosas que no funcionaron, fueron lentas, generaron preguntas redundantes, etc.
5. **Producir retro del mini-feature** (output natural de Fase 5).
6. **Consolidar retro del meta-refactor** (`001-poneglyph-5phase-workflow/retro.md`):
   - Lecciones del propio meta-refactor (US1-US9): qué fue duro, qué fue fácil.
   - Lecciones del dogfooding: qué falló empíricamente.
   - Promociones candidatas: ¿algo del meta-refactor mismo merece promoción global?
   - Living-spec deltas: ¿el spec.md original sigue describiendo lo entregado o hay deltas?
   - Commandments check: ¿alguno violado durante el camino?
   - Action items: HUs follow-up si emergen.
7. **Cerrar feature**: actualizar frontmatters a `status: closed`.
8. **Commit final**: "feat(workflow): close 001-poneglyph-5phase-workflow after dogfooding validation".

### Razón

`/flow` command se creó en US8 (al 70-80% del refactor). El meta-refactor en sí fue dogfooding masivo del PROCESO conceptual (5 fases, hard gates, drillme, decisiones absorbidas, living-spec loop) pero no del COMMAND `/flow`. Hacer dogfooding sintético sobre mini-task "documentar el workflow" hubiera sido ceremonia — el meta-refactor entero ya validó el proceso. El dogfooding real del COMMAND debe ocurrir sobre una mini-feature genuina (no sintética) en sesión siguiente.
